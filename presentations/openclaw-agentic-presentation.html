The Pi Philosophy

4 tools. Zero MCP. Maximum control.

Pi's Radical Minimalism

The Shortest System Prompt in the Industry

  • Only 4 tools: read, write, edit, bash
  • System prompt + tools < 1,000 tokens total
  • Multi-provider: Anthropic, OpenAI, Google, xAI, Groq, OpenRouter
  • Mid-session model switching with context handoff
  • No MCP — CLI tools + progressive disclosure instead
  • Top of Terminal-Bench 2.0 with Claude Opus

MCP Servers

Token-Hungry

  • Playwright MCP: 21 tools, 13,700 tokens
  • Chrome DevTools MCP: 26 tools, 18,000 tokens
  • Tool descriptions loaded every session
  • Outputs can't be piped or composed

Bash CLI Tools

225 Tokens

  • 4 scripts: start, navigate, eval, screenshot
  • README loaded only when needed
  • Composable via Unix pipes
  • Agent can write new tools on the fly

The Blueprint Engine

Deterministic wrapping non-deterministic

The Core Pattern

Deterministic [D] + Non-Deterministic [N]

minion.sh 
# [D] Pre-compute — zero tokens wasted on exploration
git diff --name-only main..HEAD > changed.txt
pnpm test --json > test-results.json

# [N] Agent works with perfect context
pi "Read context.json and implement the fix"

# [D] Deterministic quality gates
pnpm lint && pnpm test

# [N] Agent reviews its own work
pi "Review the diff for bugs"

# [D] Ship it
gh pr create --fill

The Hard Truths

What actually works and what doesn't

What Works

  • Deterministic gates catch real bugs
  • Pre-computing context eliminates waste
  • Parallel analysis (50 agents on triage)
  • In-loop steering for complex code
  • Skills the agent wrote and maintains

What Doesn't

  • Agent armies writing code unsupervised
  • Trusting agents to find ALL needed context
  • MCP servers eating 9% of your context
  • Merging code you haven't read — cognitive debt
  • Measuring productivity with vibes, not numbers

Where to Start

1. Master Pi

4 tools + extensions + skills. Learn the minimal core before adding complexity. Read Mario's blog.

2. Build CLI Tools

225-token READMEs. Not MCP servers. Agent reads the docs, invokes via bash. Composable, cheap.

3. Pre-Compute State

Run git diff, lsp, tests BEFORE the agent touches anything. Zero exploration waste.

4. Deterministic Gates

Lint, format, test, commit — these steps don't hallucinate, don't cost tokens, don't break.

5. Connect to OpenClaw

Pi's SDK powers OpenClaw. Your extensions work in both. One agent, everywhere.

6. Measure, Don't Vibe

Track cost per PR, CI pass rate, tool calls per thread. If you can't measure it, you can't improve it.

DE

Recommended Pattern

Linux + Windows Agentic Dev Environment

  • Run tmux + mprocs everywhere — bare Linux, WSL, and cloud boxes
  • Add psmux only when you need tmux-style panes in native PowerShell (no WSL)
  • Same thread/sandbox model across all environments — one muscle memory

On Linux (bare metal or WSL)

tmux + mprocs

  • sudo apt install tmux — optional gpakosz/.tmux for sane defaults
  • Auto-start in WSL: Windows Terminal profile → wsl.exe -d Ubuntu tmux new-session -A -s main
  • Install mprocs (binary or npm i -g mprocs), define mprocs.yaml per project
  • Run mprocs inside a tmux pane — dev servers, watchers, pi agents all in one view
# ~/.bashrc or WSL profile
if [ -z "$TMUX" ]; then
  tmux new-session -A -s main
fi
# mprocs.yaml
procs:
  server: npm run dev
  watch: bun --watch test
  agent: pi

On Windows (outside WSL)

psmux + mprocs

  • psmux gives tmux UX in PowerShell — sessions, panes, detach/reattach
  • Install mprocs via scoop install mprocs or npm i -g mprocs
  • Run mprocs in a psmux pane — exact same mprocs.yaml as Linux
  • For VS Code & GUI tools, connect to WSL when you need Linux paths
# PowerShell
psmux
tmux new-session -s work
# Ctrl-b %  → split pane
# Ctrl-b d  → detach
# Same mprocs.yaml works
scoop install mprocs
mprocs

Mental Model

Two Layers, Any OS

  • tmux in Linux/WSL → long-lived workspace layer (sessions, panes, SSH, remote boxes)
  • psmux on Windows → tmux UX when you stay in PowerShell/cmd (no WSL needed)
  • mprocs → project-local process orchestrator inside tmux/psmux, driven by mprocs.yaml

Map to threads: Each tmux pane = one P-thread. mprocs = the process manager inside each pane. Same muscle memory on Linux, WSL, and Windows.
